# FastAPI 写作记忆工作台界面设计文档

日期：2026-05-30

## 1. 目标

把当前 CLI 型小说创作记忆助手扩展成一个简单的 FastAPI Web 界面。界面服务于两个核心动作：

1. 记录和管理小说创作记忆。
2. 在“灵感箱”里把作者的新构思、已检索记忆和 AI 建议合并成可确认的新记忆。

本阶段不做完整小说编辑器，不保存章节正文，不做复杂多 Agent 流程。系统仍然是“创作记忆库”，而不是自动写作平台。

## 2. 推荐方案

采用“左侧导航 + 中央工作区 + 右侧相关记忆/AI 建议栏”的工作台布局。

备选方案：

- 单页表单式：最快实现，但大纲、人物、场景之间的关系不直观。
- 多页面管理后台式：分类清楚，但灵感箱工作流会被拆散，使用时需要频繁跳转。
- 工作台式：保留清晰分类，同时让灵感输入、记忆检索、AI 建议和确认保存在同一屏完成。

推荐工作台式，因为当前系统的价值在“输入构思 -> 召回记忆 -> AI 建议 -> 确认入库”的闭环，布局应优先服务这个闭环。

## 3. 信息架构

左侧导航包含：

- 灵感箱
- 大纲
- 人物
- 场景
- 剧情
- 伏笔/回调
- 世界观
- 全部记忆

顶部工具栏包含：

- 当前项目名
- 记忆总数
- 向量库同步状态
- LLM 配置状态

右侧常驻栏包含：

- 当前输入检索到的相关记忆
- AI 建议
- 待确认新增记忆

## 4. 记忆类型扩展

当前 `Memory.type` 是字符串，保存格式可以兼容扩展。界面设计将正式引入以下类型：

| 类型 | 用途 | 关键字段 |
| --- | --- | --- |
| `outline` | 大纲、卷纲、章节计划、阶段性剧情目标 | `title`, `content`, `tags` |
| `character` | 人物设定、动机、变化、关系 | `name`, `content`, `tags` |
| `scene` | 场景设定、地点、氛围、冲突、出场人物 | `title`, `content`, `tags` |
| `plot` | 剧情点、事件、转折 | `title`, `content`, `tags` |
| `callback` | 伏笔、回调、线索、悬念 | `title`, `content`, `tags` |
| `worldbuilding` | 世界观、规则、组织、历史、术语 | `title`, `content`, `tags` |
| `note` | 暂未归类的灵感或素材 | `title`, `content`, `tags` |

建议保存格式在现有字段基础上扩展：

```json
{
  "id": "uuid",
  "type": "scene",
  "title": "雨夜码头重逢",
  "name": null,
  "content": "顾北辰在雨夜码头再次看见红怀表，触发童年事故记忆。",
  "source": "用户原始输入片段",
  "tags": ["码头", "红怀表", "顾北辰"],
  "status": "confirmed",
  "created_at": "2026-05-30T12:00:00",
  "updated_at": "2026-05-30T12:00:00"
}
```

兼容策略：

- 旧记忆没有 `title/tags/status` 时正常读取。
- `character` 继续使用 `name`。
- 其他类型优先显示 `title`，没有 `title` 时显示 `content` 前 30 字。

## 5. 灵感箱工作流

灵感箱是第一屏，也是主工作流。

流程：

1. 用户输入自由构思。
2. 点击“检索记忆”后，系统用输入内容调用现有 `MemoryRetriever.search_memories()`。
3. 右侧显示相关记忆，按类型分组。
4. 点击“生成建议”后，系统把用户构思和相关记忆交给 AI。
5. AI 返回两部分：
   - 创作建议：开放式建议，不直接替作者决定。
   - 待新增记忆：结构化 JSON 候选项。
6. 用户可以编辑候选记忆的类型、标题、内容、标签。
7. 点击“确认入库”后保存到 JSON，并同步/更新向量库。

关键原则：

- AI 只能提出候选记忆，不能自动写入。
- 确认前必须允许用户编辑。
- 相关记忆必须可见，避免 AI 建议脱离已有设定。

## 6. 页面设计

### 6.1 灵感箱

左侧主区域：

- 多行输入框：写入当前构思。
- 类型提示：可选“自动判断 / 大纲 / 人物 / 场景 / 剧情 / 伏笔 / 世界观”。
- 操作按钮：检索记忆、生成建议、清空。

中间结果区域：

- AI 建议列表。
- 每条建议可一键转成候选记忆。

右侧区域：

- 相关记忆列表。
- 待确认新增记忆卡片。
- 确认入库按钮。

### 6.2 大纲

用途：管理故事宏观结构。

视图：

- 大纲记忆列表。
- 按“主线 / 支线 / 卷 / 章节 / 阶段目标”标签筛选。
- 点击条目后在右侧显示详情和相关人物/场景。

### 6.3 人物

用途：管理人物设定。

视图：

- 人物列表，按 `name` 聚合。
- 人物详情显示所有相关 character 记忆。
- 允许从人物页发起“基于此人物生成建议”。

### 6.4 场景

用途：管理地点、情境、冲突和氛围。

视图：

- 场景卡片列表。
- 每张卡显示标题、摘要、标签、最近更新时间。
- 可筛选包含某人物或某地点的场景。

### 6.5 全部记忆

用途：兜底管理和检索。

视图：

- 类型筛选。
- 关键词搜索。
- 编辑、删除、导出。

删除需要二次确认。

## 7. 后端接口草案

后续实现时，FastAPI 可先提供这些路由：

```text
GET  /                       灵感箱首页
GET  /memories               全部记忆
GET  /memories/{type}        某类记忆页
POST /api/memories/search    检索相关记忆
POST /api/inspirations/advise 生成建议和候选记忆
POST /api/memories           确认新增记忆
PUT  /api/memories/{id}      更新记忆
DELETE /api/memories/{id}    删除记忆
POST /api/vector/sync        同步向量库
```

HTML 首版建议使用 FastAPI + Jinja2 模板 + 少量原生 JavaScript，不引入前端框架。这样改动范围小，也更贴合当前 Python 项目。

## 8. AI 输出格式

灵感箱里的 AI 输出建议拆成两个结构：

```json
{
  "suggestions": [
    "可以把红怀表作为顾北辰童年事故的触发物。",
    "码头场景适合保留潮湿、噪声和闪回三类感官线索。"
  ],
  "candidate_memories": [
    {
      "type": "scene",
      "title": "雨夜码头触发回忆",
      "content": "顾北辰在雨夜码头看到红怀表，引发童年事故相关记忆。",
      "tags": ["顾北辰", "码头", "红怀表"]
    }
  ]
}
```

如果 AI 返回格式不合法，界面显示原始文本，并提示用户手动转成记忆。

## 9. 错误处理

- LLM 未配置：顶部显示“LLM 未配置”，生成建议按钮禁用或点击后显示配置提示。
- 向量库未初始化：允许普通记忆列表加载，检索时提示先同步。
- Embedding 模型加载慢：检索按钮显示加载状态，避免界面无反馈。
- AI JSON 解析失败：保留建议文本，不自动入库。
- 保存失败：候选记忆保留在页面，不清空用户输入。

## 10. 实现边界

本阶段做：

- FastAPI Web 入口。
- 基础模板和页面布局。
- 记忆类型扩展。
- 灵感箱检索、建议、候选确认流程。
- 大纲、人物、场景等分类视图。

本阶段不做：

- 用户登录。
- 多项目切换。
- 富文本章节编辑器。
- 自动生成整章正文。
- 复杂关系图谱。

## 11. 验收标准

完成后应满足：

1. 启动 FastAPI 后能打开灵感箱首页。
2. 可以输入构思并检索相关记忆。
3. 可以生成 AI 建议和候选记忆。
4. 候选记忆确认后写入 `data/memories/*.json`。
5. 新类型记忆能在“大纲 / 人物 / 场景”等页面中显示。
6. 旧的 `character / plot / callback` 记忆仍能正常读取。
